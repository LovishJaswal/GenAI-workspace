## Fine Tuning

**Fine Tuning** is the process of further training a pre-trained AI model for a specific task by updating its parameters.

## RAG (Retrieval-Augmented Generation)

**RAG** is a technique that retrieves relevant external information before an LLM generates a response.

> **Note:** RAG does **not** modify the model's knowledge. It simply provides additional context to the model during inference.

# LLM Memory

Large Language Models are **stateless**.

They do **not** remember previous conversations on their own.

Applications such as ChatGPT maintain conversation history by sending previous messages along with each new prompt.

This creates the illusion that the model has memory.

# Embeddings

Embeddings are numerical vector representations of data (such as text, images, or audio) that capture their semantic meaning.
Similar pieces of information have embeddings that are close to each other in vector space.

### Example

```text
"Dog"  → [0.23, -0.91, 0.45, ...]
"Puppy" → [0.25, -0.88, 0.43, ...]
```

Since **Dog** and **Puppy** have similar meanings, their embeddings are close together.

---

# Cosine Similarity

Cosine Similarity measures the similarity between two embeddings.

- Value close to **1** → Very similar
- Value close to **0** → Unrelated
- Value close to **-1** → Opposite direction (rare in most embedding models)

It is commonly used in Retrieval-Augmented Generation (RAG) to find the most relevant documents.

# GGUF

**GGUF (GPT-Generated Unified Format)** is a file format used to store AI model weights for efficient local inference.

Its purpose is to make models easier and faster to run on personal computers.

### Benefits

- Faster loading
- Lower RAM usage
- Works well with CPU inference
- Supported by Ollama and llama.cpp

> **Note:** GGUF is **not** an AI model. It is only a file format.

# Why Local Models?

Modern AI models such as GPT, Llama, Gemma, and Qwen contain **billions of parameters**, making them extremely large and computationally expensive to run.

Initially, these models were hosted on powerful cloud servers. Users accessed them through APIs, which required:

- Internet connectivity
- Paid API usage
- Sending data to external servers (raising privacy concerns)

Running models locally solves many of these issues.

### Advantages of Local Models

- Better privacy (data remains on your device)
- No internet required after downloading
- No API costs
- Faster responses for many use cases
- Full control over the model

However, original models require a large amount of RAM or VRAM, making them difficult to run on most personal computers.

---

# Quantization

**Quantization** is the process of reducing the precision of a model's weights to decrease its size and improve inference speed.

### Example

```text
Original value : 3.14159265
Quantized value: 3.14
```

This significantly reduces memory usage while causing only a small decrease in model quality.

### Benefits

- Smaller model size
- Faster inference
- Lower RAM/VRAM requirements
- Enables local execution on consumer hardware

---

# GGUF

To efficiently store and run quantized models, formats such as **GGUF** were introduced.

GGUF packages the model's weights in a format optimized for local inference.

Tools like **Ollama** and **llama.cpp** can directly load GGUF models.

# Understanding Models and GGUF

A common misconception is:

> **"GGUF is the model."**

This is **incorrect**.

A model consists of:

- Architecture
- Weights

GGUF is **only a file format** used to store the weights efficiently.

For example:

```text
Llama
│
├── safetensors
├── .bin
├── GGUF
└── Other formats
```

The same applies to:

- Mistral
- Gemma
- Qwen
- Phi
- DeepSeek

Most model creators release checkpoints in formats like **safetensors**.

The community often converts them into **GGUF** for easier local inference.

> **Important:** The "GPT" in **GGUF** does **not** mean it only supports GPT models.

---

# GGML → GGUF

Before GGUF, the llama.cpp ecosystem used another format called **GGML**.

GGUF replaced GGML with a more flexible and feature-rich format while maintaining compatibility with the existing ecosystem.

# Stateless Nature of Large Language Models (LLMs)

Large Language Models (LLMs) are **stateless**, meaning they do **not** remember previous conversations by themselves.

Every request sent to an LLM is processed independently.

---

## How ChatGPT Remembers Conversations

Although LLMs are stateless, applications such as ChatGPT appear to have memory because the application itself manages the conversation history.

The workflow is as follows:

```text
User Message
      │
      ▼
Application (ChatGPT)
      │
      ├── Retrieves previous conversation
      ├── Combines it with the new message
      ▼
Creates a single prompt
      │
      ▼
LLM
      │
      ▼
Generates Response
```

The model does **not** remember earlier messages on its own. Instead, the application sends previous messages along with every new request.

---

# Context Window

The **Context Window** is the maximum number of tokens an LLM can process in a single request.

It includes:

- System Prompt
- Conversation History
- User Prompt
- Retrieved Documents (RAG)
- Model Response

All of these consume tokens from the available context window.

---

# Why Context Management is Important

As conversations become longer, more tokens are required to send previous messages back to the model.

This can lead to:

- Increased latency
- Higher API cost
- Larger memory usage
- Exceeding the model's context window

Therefore, applications must manage conversation history efficiently.

---

# Common Context Management Techniques

## 1. Truncation

The oldest messages are removed once the context becomes too large.

```text
Before

Message 1
Message 2
Message 3
Message 4
Message 5

↓

After

Message 3
Message 4
Message 5
```

---

## 2. FIFO (First In, First Out)

The earliest conversation is discarded first.

```text
Oldest ─────────────► Newest

A → B → C → D → E

Remove A first.
```

---

## 3. LRU (Least Recently Used)

Instead of removing the oldest message, the least recently referenced information is removed.

This technique is commonly used in caching systems and can also be applied to conversation management.

---

## 4. Summarization

Instead of keeping every previous message, the application summarizes older conversations into a shorter version.

Example:

```text
Original Conversation

• User likes Python.
• User is learning RAG.
• User uses Ollama.
• User prefers local LLMs.

↓

Summary

User is learning GenAI using local LLMs with Ollama and is currently studying RAG.
```

This preserves important information while using fewer tokens.

```text
User
   │
   ▼

   │
   ├── Retrieves Conversation History
   ├── Retrieves Documents (Optional RAG)
   ├── Adds System Prompt
   ▼

   │
   ▼

   │
   ▼

```

# Key Takeaways

- LLMs are **stateless**.
- Applications maintain conversation history, not the model.
- Every request contains all necessary context.
- Tokens include both the input and the generated output.
- Long conversations require context management.
- Common techniques include:
  - Truncation
  - FIFO
  - LRU
  - Summarization
